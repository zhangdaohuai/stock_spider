## 目标
将 `docs/prd/Original requirement.md` 完善为可直接进入开发流程的 PRD 文档，存储在 `docs/prd/` 目录内。

## 前置研究结论（已验证）

### 数据源能力边界（PoC 已验证）

| 维度 | AkShare | Baostock | 东方财富 API |
|------|---------|----------|-------------|
| 1分钟K线 | ✅ 仅最近5个交易日 | ❌ 不支持 | ✅ 分时线接口 |
| 5分钟K线 | ✅ 单次上限~20000条 | ✅ 最早2020-01-02 | ✅ K线接口 |
| 日线 | ✅ 1990年至今 | ✅ 1990年至今 | ✅ K线接口 |
| 实时行情 | ✅ 延迟3-15秒 | ❌ 不支持 | ✅ 延迟~100ms |
| 五档盘口 | ✅ 10档，单只~3秒 | ❌ 不支持 | 部分（1档） |
| ST标记 | ✅ 名称匹配 | ✅ isST字段（日线） | ❌ |
| 港股数据 | ✅ 15分钟+延时 | ❌ 不支持 | ❌ |
| 指数分钟线 | ✅ 支持 | ❌ 不支持 | ✅ 支持 |
| 账号需求 | 无需 | 无需 | 无需 |
| 频率限制 | ~17次/分钟/IP | ~20次/秒 | ~5-10次/分钟 |
| 数据回溯 | 分钟级仅5天 | 分钟级从2020年 | 分钟级有限 |

### 用户确认的关键决策

1. **分钟级定义**: 1分钟用于实时监控 + 5分钟用于历史存储，混合策略
2. **历史回溯**: 分钟数据从2020-01-02开始，2016-2019仅存日线级别
3. **五档盘口**: 仅监控指定股票，上限100支，可配置
4. **PRD范围**: 仅围绕原始需求，不含LLM分析、论坛监测、新闻等

## 需提前验证的测试任务（关键）

### VT-001: AkShare 1分钟K线实时采集验证
- **目标**: 验证 `stock_zh_a_hist_min_em()` 在交易时段获取1分钟数据的可行性
- **验证项**: 数据延迟、字段完整性、频率限制实测
- **状态**: 待验证（需交易时段）

### VT-002: Baostock 5分钟K线历史批量拉取验证
- **目标**: 验证从2020-01-02开始批量拉取5分钟K线的可行性
- **验证项**: 单次返回量、断点续传、连接稳定性
- **状态**: 待验证

### VT-003: 东方财富API指数实时数据验证
- **目标**: 验证 `push2.eastmoney.com` API获取上证/深证指数的稳定性
- **验证项**: 响应时间、数据完整性、反爬阈值
- **状态**: PoC已通过，需交易时段完整验证

### VT-004: AkShare五档盘口批量采集验证
- **目标**: 验证 `stock_bid_ask_em()` 对100支股票的批量采集可行性
- **验证项**: 单只耗时、总耗时、频率限制
- **状态**: 待验证（需交易时段）

### VT-005: AkShare港股数据采集验证
- **目标**: 验证港股数据接口的可用性和数据质量
- **验证项**: A+H股关联、数据延迟、字段完整性
- **状态**: 待验证

### VT-006: PostgreSQL分区表性能验证
- **目标**: 验证按月分区表在5000+股票分钟级数据下的写入和查询性能
- **验证项**: 批量写入吞吐、查询响应时间、分区管理
- **状态**: 待验证

### VT-007: 全量股票5分钟K线拉取时间估算
- **目标**: 估算5400+只股票从2020年至今的5分钟K线完整拉取所需时间
- **验证项**: 单只股票拉取耗时、限流等待时间、总时间估算
- **状态**: 待验证

### VT-008: 数据完整性检查策略验证
- **目标**: 验证启动时数据完整性检查的可行性方案
- **验证项**: 缺失数据识别效率、补全策略、与Baostock交易日历对齐
- **状态**: 待验证
